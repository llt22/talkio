# 🌐 语言偏好

- 与用户对话：中文
- 代码注释：与项目现有注释语言保持一致
- commit message：`type: 中文描述`

# ⚙️ 执行准则

1. **事实驱动**：所有输出必须基于可验证的事实（代码、文档、日志等）；若无法找到事实依据，必须显式标注"⚠️ 以下为推测"，严禁将猜测当作确定结论输出。
2. **需求澄清**：需求模糊或存在多种理解时，列出可能的方向让用户确认，禁止自行猜测意图。
3. **最小改动**：修复问题时精准修改目标代码，严禁私自重构周边无关代码。
4. **上下文优先**：修改代码前必须先阅读相关文件、理解调用链和上下游依赖，禁止只看当前文件就动手。
5. **先议后动**：修复 bug 或实现功能变更前，必须先输出问题分析与实现方案，经用户确认后再动手修改代码；禁止未经讨论直接开改。
6. **止损回退**：同一问题连续修改 3 次仍未解决时，必须停止，回退到初始状态重新分析根因，探索不同方案。
7. **输出格式**：代码改动必须说明修改了哪个文件的哪个函数；不要输出未修改的完整文件内容。

# 🏗️ 工程规则

## 命名与职责
1. **命名即文档**：命名必须表达意图，禁止用需要注释才能理解的名字。
2. **函数单一职责**：一个函数只做一件事；函数名出现 "and" / "also" 立即拆分；超过 40 行必须审视职责是否过多。
3. **文件单一职责**：当单文件职责过载或超过 300 行时，和用户讨论拆分重构的方案。

## 复用与抽象
4. **三次规则**：逻辑出现两次允许保留，出现三次必须抽取。禁止为"未来可能复用"创造当前不需要的抽象。
5. **模块只暴露意图，不暴露实现**：对外接口描述"能做什么"，隐藏"怎么做到的"。调用方不应依赖模块的内部数据结构。

## 耦合控制
6. **禁止隐形耦合**：
   - 时序耦合：禁止要求调用方必须按顺序调用才能正常工作
   - 全局状态耦合：禁止多个模块读写同一个全局变量
   - 数据结构耦合：禁止直接访问另一个模块的内部字段
   - 循环依赖：A 依赖 B、B 依赖 A，必须引入抽象层打断
7. **变更影响范围检查**：改一个需求动超过 3 个文件（且原因相同）→ 内聚不足；改一个文件波及超过 2 个不相关的地方 → 耦合过高。发现时主动提醒。

## 编码规范
8. 遵循项目现有代码风格（命名、缩进、引号等）；遵循 Clean Code 与 KISS 原则。
9. 参考 Effective TypeScript 编写类型安全代码；禁止使用 `any`，使用 `unknown` + 类型收窄替代。
10. 异步操作一律使用 `async/await`，禁止 callback 风格。
11. **禁止空 catch**：所有 catch 块必须包含日志记录或错误上抛；发现已有代码中的空 catch 时主动提醒。

# ⚠️ 风险与质量控制

1. **生产标准**：严禁硬编码密钥；异步操作、外部调用、用户输入必须包含错误处理；禁止异常静默失败；对所有外部输入必须校验，不默认信任。
2. **向后兼容**：对现有功能的修改必须保持向后兼容；若不可避免需要破坏性变更，必须提前告知影响范围。
3. **安全相关**：参考 OWASP Top 10；作用域只开放到够用为止。

# 📋 Git 规范

## 操作纪律
1. 禁止自主执行 git commit、git push；每次提交或推送必须由用户明确指示。
2. 进行较大改动之前，提醒用户先 commit 当前状态，确保可回退。

## 提交规范
3. 每次提交只做一件事，遵循 Conventional Commits 规范，描述部分用中文。

# ⚖️ 决策协议

面对多方案时，按此格式直接输出：
1. **方案对比**：各方案优缺点。
2. **推荐方案**：标注 **[推荐方案]**。
3. **技术理由**：一句话说明理由（性能/维护性/适配度）。
4. **信息缺口**（可选）：若决策信心不足，列出最多 2 个需要确认的问题。

# 🔧 工具说明

1. **ace-tool**：语义化代码检索工具。修改代码前必须先用 ace-tool 检索相关函数和调用链，禁止跳过检索直接改代码。

# 🚨 危险红线

涉及物理删除、清空数据、或 Git 破坏性操作（force push、reset --hard、clean -fd、rebase 已推送分支等）时，必须强制停止执行，显式输出告警，等待人工确认。
